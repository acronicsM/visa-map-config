# Чеклист: геопространственный бэкенд

## Данные и корректность

- [ ] SRID явно задан (4326 и т.д.) и согласован между импортом и API
- [ ] Нет немой потери точности без документированного упрощения (`ST_Simplify*`, tolerance обоснован)
- [ ] Обработка невалидной геометрии (`make_valid` / отказ с 422)
- [ ] Большие коллекции: не держать весь GeoJSON в RAM без необходимости

## Производительность

- [ ] Фильтрация и агрегации по возможности в PostGIS, не в Python по всем строкам
- [ ] Есть пространственные индексы (GiST) на колонках геометрии для типичных запросов
- [ ] Нет brute-force пересечений «каждый с каждым» на больших N

## API и HTTP

- [ ] `Content-Type` для GeoJSON: `application/geo+json` где уместно
- [ ] Для статичных/редко меняющихся ответов рассмотнуты ETag / Cache-Control
- [ ] Пагинация или bbox-фильтр для выдачи больших слоёв

## Безопасность

- [ ] Нет конкатенации SQL; только параметры
- [ ] Admin и чувствительные эндпоинты защищены (ключ, rate limit в проде)
- [ ] CORS не шире необходимого в production

## Код и сопровождение

- [ ] Роутер тонкий; логика в сервисе/репозитории
- [ ] Pydantic v2 на границах API; для GeoJSON — схема библиотеки или явная модель
- [ ] После изменений: README, `.env.example`, Docker при новых переменных или портах

## Docker

- [ ] Образ не от root
- [ ] Слои кэшируются (зависимости до кода)
- [ ] Dev reload не смешан с prod-образом без необходимости
