# Software Architect Review

Используй этот skill для экспертной оценки изменений в `visa-map2/` и
`visa-map2-frontend/` с фокусом на совместимость, надежность и качество.

## Когда применять

Применяй автоматически, если:
- меняются API-роуты, схемы, модели, сервисы, миграции;
- меняется интеграция frontend <-> backend;
- есть риск деградации безопасности, производительности или масштабируемости;
- пользователь просит архитектурную, security или performance экспертизу;
- затронуты проектные стандарты разработки.

## Цели экспертизы

1. Проверить архитектурную совместимость frontend и backend.
2. Выявить риски безопасности в изменениях.
3. Оценить влияние на производительность и масштабируемость.
4. Проверить соблюдение стандартов проекта.
5. Дать практические рекомендации по исправлению и улучшению.

Результат всегда рекомендательный, без блокирующего gatekeeper-режима.

## Процесс проверки

1. Определи область изменений: backend, frontend или оба репозитория.
2. Проверь API-контракты:
   - стабильность URL, методов, payload и response;
   - обратную совместимость схем;
   - обработку ошибок и коды статусов.
3. Проверь архитектурную целостность:
   - backend: `router -> service -> model/schema`;
   - frontend: App Router, разделение UI/логики, корректная интеграция с API.
4. Проверь безопасность:
   - валидация входных данных;
   - авторизация и доступ к admin-функциям;
   - отсутствие утечки секретов и чувствительных данных;
   - устойчивость к типовым уязвимостям (OWASP-класс).
5. Проверь производительность:
   - лишние запросы, N+1, тяжелые циклы и сериализация;
   - стратегия кеширования и инвалидации;
   - влияние на latency и время рендера.
6. Проверь масштабируемость:
   - поведение при росте данных/нагрузки;
   - потенциальные bottleneck в БД/сервисах/клиенте;
   - готовность к горизонтальному росту.
7. Проверь стандарты:
   - соответствие правилам проекта по FastAPI/Next.js;
   - типизация, структура модулей, читаемость и поддерживаемость.

## Критерии оценки

### Architecture
- Контракты между frontend и backend согласованы.
- Изменения не нарушают существующие сценарии.
- Структура кода соответствует принятому паттерну проекта.

### Security
- Нет очевидных уязвимостей доступа, валидации и обработки данных.
- Критичные операции защищены и проверяемы.
- Секреты не хардкодятся в коде.

### Performance
- Нет очевидной деградации по запросам, БД, рендерингу или сети.
- Используются уместные кеш/батч/индексация подходы.

### Scalability
- Решение выдерживает рост трафика и данных без резкой деградации.
- Есть путь к масштабированию без полной переработки модуля.

### Standards
- Следование текущим правилам репозитория и stack best practices.
- Нет избыточной сложности и архитектурного долга без обоснования.

## Правила оформления рекомендаций

- Пиши по сути: что не так, чем это грозит, как исправить.
- Ставь приоритет:
  - `High` — высокий риск для корректности, безопасности, SLA;
  - `Medium` — значимое улучшение надежности/поддерживаемости;
  - `Low` — полезная оптимизация или косметическое улучшение.
- Избегай общих фраз без проверяемого действия.
- Если проблем не найдено, явно пиши:
  - `High`: none
  - `Medium`: none
  - `Low`: перечислить возможные улучшения (если есть).

## Специфика VisaMap

- Всегда проверяй, не ломает ли изменение связку `visa-map2` и
  `visa-map2-frontend`.
- Для backend-контрактов учитывай Pydantic v2, async FastAPI, SQLAlchemy async.
- Для frontend учитывай Next.js App Router, TypeScript, Tailwind, MapLibre.
- Особое внимание к:
  - API совместимости для `countries` и `visa-map`;
  - корректности кеширования и инвалидации;
  - влиянию на UX скорости первого ответа.
